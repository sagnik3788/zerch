## рҹ“Ң Overview

**Zerch** is a **semantic search engine purpose-built for log files**. Traditional log search tools rely on exact keyword matching вҖ” you get results only when you type the exact string that appears in the log. Zerch is fundamentally different.

Instead of keyword matching, Zerch **embeds every log line into a 384-dimensional semantic vector** using a locally-running sentence transformer model (`all-MiniLM-L6-v2` via ONNX Runtime). These vectors are stored in a compact, append-only binary file. When you search, your natural language query is embedded in real-time and compared against every stored vector using **cosine similarity** вҖ” returning the logs that are *semantically closest* to your intent, not just lexically similar.

On top of search, Zerch integrates with the **Groq API** to provide AI-powered log summarization: click any search result and get an instant LLM-generated explanation and fix suggestion.

**Key capabilities at a glance:**

| Capability | Description |
|---|---|
| рҹ”Қ **Semantic Search** | Find logs by meaning, not just keywords |
| рҹ§  **Local Embeddings** | All-MiniLM-L6-v2 runs 100% on your machine |
| вҡЎ **Rust Performance** | Zero-copy binary I/O, memory-safe, blazing fast |
| рҹӨ– **AI Summarization** | Groq-powered LLM insights on any log entry |
| рҹ–ҘпёҸ **CLI Tool** | Index and search logs directly from the terminal |
| рҹ“Ў **MCP Server** | Plug Zerch into any AI agent via Model Context Protocol |
| рҹҗі **Docker Ready** | Full containerized deployment with one command |

## вңЁ Features

### рҹ”Қ Semantic Understanding
Type what you *mean*, not what you remember. Search for `"database connection refused"` and find logs mentioning `"ECONNREFUSED"`, `"timeout on port 5432"`, or `"postgres unreachable"` вҖ” because Zerch understands context.

### вҡЎ Sub-Millisecond Search
The Rust backend reads vectors directly from disk using zero-copy byte reinterpretation, computes cosine similarity for every record in microseconds, and returns ranked results instantly.

### рҹ“Ө Drag-and-Drop Log Ingestion
Upload any `.log`, `.txt`, or `.json` file via the browser UI. Every non-empty line is parsed, embedded, and stored вҖ” with metadata extraction (log level, service name, timestamp) happening automatically.

### рҹӨ– AI-Powered Summarization
Click any search result to send the raw log text to the Groq API. The LLM analyzes the log, explains what happened in plain English, and suggests a fix вҖ” displayed live in the right sidebar.

### рҹ–ҘпёҸ CLI Engine
Prefer the terminal? Use `zerch-engine` to index entire log files and run semantic searches without touching a browser.

### рҹ“Ў MCP Protocol Support
The `zerch-mcp-server` exposes `embed_text` and `search_logs` as MCP tools over **stdio** or **HTTP**, making Zerch a first-class citizen in any AI agent workflow (Claude Desktop, LangChain, custom agents).

### рҹҸ—пёҸ Modular Workspace Architecture
Six independent Rust crates, each with a single responsibility вҖ” core math, embedding, storage, CLI engine, HTTP API, and MCP server. Add features without touching unrelated code.

## рҹ“Ұ Binary Storage Format

Every embedded log line is serialized into `zerch_data.bin` using a flat, append-only binary format with zero serialization overhead:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  vec_len (u32)   в”Ӯ  vector data             в”Ӯ  text_len (u32)  в”Ӯ  text (UTF-8)    в”Ӯ
в”Ӯ  4 bytes LE      в”Ӯ  vec_len Г— 4 bytes (f32) в”Ӯ  4 bytes LE      в”Ӯ  text_len bytes  в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

For `all-MiniLM-L6-v2`, `vec_len` is always **384**, making each record:
- **Fixed:** `4 + (384 Г— 4) + 4` = **1,544 bytes** of header + vector
- **Variable:** `text_len` bytes of raw UTF-8 log text

Vectors are written using `unsafe { std::slice::from_raw_parts(...) }` for **zero-copy** byte reinterpretation вҖ” no serialization library, no overhead.

## рҹ“– How It Works

### 1. Indexing вҖ” Upload вҶ’ Embed вҶ’ Store

```
User uploads log.txt
        в”Ӯ
        в–ј
  Actix-Web parses multipart
        в”Ӯ
        в–ј
  For each non-empty line:
    в”ңв”Җв”Җ extract_log_level()   вҶ’ ERROR / WARN / INFO / DEBUG
    в”ңв”Җв”Җ extract_service()     вҶ’ text between first [...]
    в”ңв”Җв”Җ extract_timestamp()   вҶ’ current datetime
    в””в”Җв”Җ LocalEmbedder.embed(line)
            в”Ӯ
            в”ңв”Җв”Җ HF Tokenizer вҶ’ input_ids, attention_mask, token_type_ids
            в”ңв”Җв”Җ Truncate to 512 tokens
            в”ңв”Җв”Җ ONNX Runtime inference вҶ’ last_hidden_state [1, seq, 384]
            в”ңв”Җв”Җ Mean pooling over seq dimension вҶ’ [384]
            в””в”Җв”Җ L2 normalization вҶ’ unit vector [384]
            в”Ӯ
            в–ј
  VectorStore.append_vector(vector, text)
  вҶ’ write to zerch_data.bin (binary, append-only)
        в”Ӯ
        в–ј

```

### 2. Searching вҖ” Query вҶ’ Embed вҶ’ Rank

```
User types: "why is the database failing?"
        в”Ӯ
        в–ј

        в”Ӯ
        в–ј

        в”Ӯ
        в–ј

    For each stored record:
```

## рҹ”§ Crate Deep Dive

### `zerch-core` вҖ” Pure Math, Zero Dependencies

The mathematical foundation of Zerch. No external crates вҖ” just clean Rust.

```rust
// Cosine similarity between two embedding vectors
let sim = cosine_similarity(&query_vector, &log_vector);
println!("Score: {:.4}", sim.score);  // Range: [-1.0, 1.0]

// Euclidean distance (alternative metric)
let dist = euclidean_distance(&vec_a, &vec_b);
println!("Distance: {:.4}", dist.distance);
```

Both functions return sentinel values (`f32::NEG_INFINITY` / `f32::INFINITY`) on dimension mismatch.

### `zerch-embed` вҖ” Local Inference Pipeline

```
Input text
  вҶ’ HuggingFace Tokenizer (WordPiece, max 512 tokens)
  вҶ’ ONNX Runtime Session (CUDAExecutionProvider вҶ’ CPU fallback)
  вҶ’ last_hidden_state: shape [1, seq_len, 384]
  вҶ’ Mean pooling across seq dimension вҶ’ [384]
  вҶ’ L2 normalization вҶ’ unit vector [384]
  вҶ’ Vec<f32>
```

The model (`all-MiniLM-L6-v2`) is downloaded once from HuggingFace Hub and cached in `~/.cache/huggingface/`.

### `zerch-storage` вҖ” Zero-Copy Binary I/O

```rust
let store = VectorStore::new("zerch_data.bin");

// Zero-copy write: reinterprets &[f32] as &[u8] via unsafe ptr cast
store.append_vector(&vector, "ERROR: connection refused")?;

// Clear the store (truncates the file)
store.clear()?;
```

| Limitation | Status |
|---|---|

| **Timestamp parsing** | `extract_timestamp()` always returns current time, not the log's own timestamp |
